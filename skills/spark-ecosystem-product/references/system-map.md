# Spark Ecosystem Product Reference

## Core System Map

### Spark Intelligence

Owns:

- persistent agent product shell
- gateway
- adapters
- session and identity mapping
- runtime orchestration
- chip and path attachment
- true scheduled work through the shared job harness

### Spark Researcher

Owns:

- core runtime intelligence
- reasoning loop
- execution planning
- base agent continuity
- governing-loop execution shape

### Spark Swarm

Owns:

- deep delegation
- parallel specialist execution
- multi-agent merge behavior

### Domain Chips

Own:

- specialized heuristics
- domain-specific workflows
- scoped intelligence surfaces

### Specialization Paths

Own:

- long-term agent growth direction
- capability deepening over time

### Memory Chip

Owns:

- memory promotion
- retrieval logic
- memory hygiene
- contradiction-aware memory policy

## Product Questions

For every product decision, answer:

1. Who owns this behavior?
2. Does it deepen the wedge?
3. Does it improve retention or only increase surface area?
4. Does it fit the lego-piece architecture?
5. Does it make the user experience feel like one coherent evolving agent?
6. Is this actually scheduled work, or am I incorrectly turning Spark logic into cron?

## Anti-Patterns

- Spark Intelligence swallowing the rest of the ecosystem
- domain chips reduced to branding only
- specialization paths that do not actually change behavior
- product ideas that fragment the one-agent identity
- turning governing-loop behavior into adapter-owned cron jobs
